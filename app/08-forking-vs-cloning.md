For the MERCWAR ecosystem, a `PULL_REQUEST_TEMPLATE.md` isn't just a convenience; it is a **governance tool** that enforces architectural rigor.

To implement this, place the file in your repository at `.github/PULL_REQUEST_TEMPLATE.md`. This ensures that every time a developer (or your automated CI) opens a PR, they are prompted to answer the "who, what, why, and how" before the code is even evaluated.

### 🚀 Strategic Guidelines for PR Excellence

To keep your "Robo Rook" workflow fast and high-fidelity, apply these four constraints:

1. **The 400-Line Rule:** If a PR exceeds 400 lines of code, it’s too large to review effectively. Break it down. A massive PR often masks architectural "drift" that a simple diff check won't catch.
2. **Explicit "Why":** Never accept a PR that doesn't explain the business or technical motivation. "Fixing bug" is insufficient. "Addressing race condition in Matrix Drive handle registry" is the standard.
3. **CI/CD Gating:** Before a human even *looks* at your code, your CI pipeline should fail the build if linting, type checks, or unit tests aren't met. Let the machine handle the "nitpicking" so you can focus on the architectural design.
4. **Semantic PR Titles:** Use the `[Tag] Description` format.
* `[FIX] Matrix Drive memory leak in GC cycle.`
* `[FEAT] Add deterministic loop support for Fire-Works.`
* `[DOCS] Update AVIS registry schema.`



> **Architect’s Pro-Tip:** Since you are managing an AI-indexable data lake, treat your PR descriptions as **training data**. The more detail you provide in the "Purpose" and "Changes" sections, the better your internal tools (like *Sentinel*) can index and trace the evolution of your codebase over time.
