# Issues, Discussions & Wikis

Within the GitHub ecosystem, **Issues**, **Discussions**, and **Wikis** form the "Non-Code Layer." While the repository stores the executable logic, these components capture the *intent, reasoning, and context* that keep a professional project sustainable over time.

---

### 📋 Issues: The Operational Tracker

Issues are the primary interface for tracking work-in-progress, bugs, and feature requests. They function as a lightweight project management tool directly integrated with your codebase.

* **Mechanics:** Each issue is an independent thread with a unique ID (e.g., `#42`).
* **Workflow Integration:** You can link issues to PRs by using keywords like "Closes #42" or "Fixes #42." When the PR is merged, GitHub automatically closes the issue, creating a clean link between the problem and the solution.
* **Best Practice:** Use **Labels** (e.g., `bug`, `feature`, `documentation`) and **Milestones** (e.g., `v1.0-Release`) to organize your backlog. For the *Sentinel* kernel, issues should be used to log runtime failures that require architectural intervention.

---

### 💬 Discussions: The Knowledge Exchange

Discussions are designed for open-ended conversation, brainstorming, and community support—conversations that don't necessarily map to a specific "to-do" item.

* **Mechanics:** Unlike issues, discussions are not meant to be "closed." They remain active as long as the conversation is relevant.
* **Use Cases:**
* **Architecture Brainstorming:** Proposing a change to the `GF` object structure and getting community feedback before writing a single line of code.
* **Q&A:** A dedicated space for users to ask how to use your *Matrix Drive* tools.
* **Announcements:** Sharing updates on the *Fire-Works* project roadmap.


* **Value:** It prevents your issue tracker from becoming cluttered with general questions, keeping the "Issues" tab focused on actionable tasks.

---

### 📖 Wikis: The Architectural Repository

Wikis provide a long-form documentation space that exists alongside your code. While `README.md` is for the "elevator pitch," the Wiki is for the "manual."

* **Mechanics:** Each repository gets its own Wiki, which is itself a Git repository. This means you can track the history of your documentation just like your source code.
* **When to use:**
* **Detailed Specs:** Deep-dive documentation into your deterministic execution models.
* **API Documentation:** Explaining the parameters and return types for your C-based internal libraries.
* **Developer Guides:** Step-by-step instructions for setting up the build environment on different OS platforms.


* **Best Practice:** Keep the Wiki clean. If documentation is needed to run the project, keep it in the main repo (`/docs`). If it is background reading or historical architecture analysis, put it in the Wiki.

---

### ⚖️ The "Robo Rook" Strategy

In a professional architecture like yours, these tools should be utilized as a **tripartite system**:

1. **Issues:** If it is broken or needs to be built, it is an **Issue**.
2. **Discussions:** If you are unsure *how* to build it or need to explain the "Why," it is a **Discussion**.
3. **Wiki:** Once the "How" and "Why" are finalized, it becomes part of the **Wiki** (or the project documentation).

> **Architect’s Pro-Tip:** Since you are building the `AVIS-DATALAKE`, consider creating a standardized template for your Issues. A well-structured issue report (e.g., "Expected Behavior" vs. "Actual Behavior") will save you hours of debugging time by ensuring every bug report provides the necessary telemetry data.
